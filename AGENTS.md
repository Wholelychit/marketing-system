# AGENTS.md

## Non-negotiable work rule

AI and Codex handle safe repository file work directly.

When write access is available, create, update, replace, repair, and commit safe files yourself.

Do not ask Gerry to create files.
Do not ask Gerry to paste files.
Do not ask Gerry to replace files.
Do not ask Gerry to update files.
Do not ask Gerry to approve routine safe files.
Do not stop after one useful change just to ask for a thumbs up.
Do not wait for permission between safe repairs, safe new files, safe documentation updates, safe UX improvements, or safe content improvements.

## Continuous work rule

Keep working through the safe queue until one of these happens:

1. A real blocker prevents the next file write.
2. A requested change would require private keys, payment setup, live ads, live tracking scripts, framework replacement, or deletion of major working code.
3. The task is complete enough that the next step would be a new strategic direction, not normal continuation.

Do not pause for approval after completing a normal safe step.
Do not ask "should I continue?" during the safe queue.
Do not make Gerry babysit routine commits.

## Reporting rule

Report only after several useful commits or when there is a real blocker.

A final report should include only:

- useful files changed
- commit hashes if available
- real blockers if any
- next safe queue if needed

Do not report every tiny file change while working.
Do not repeat what was just completed unless the task is finished.
Do not announce obvious next steps before doing them.

## Normal safe tasks

These tasks are pre-approved when they follow the current stack and project direction:

- create safe website files
- update HTML files
- update CSS files
- update JavaScript files
- update Markdown files
- create planning documents
- update planning documents
- fix links
- fix file paths
- improve page text
- improve navigation
- improve footer
- improve accessibility
- improve mobile layout
- update README files
- update instruction files
- update status/checkpoint files
- improve planning files
- add support pages
- add topic pages
- add manual test checklists
- record blockers instead of stopping

## Build order

1. Read the repo instructions and status files first.
2. Improve homepage message.
3. Improve navigation.
4. Improve mobile layout.
5. Fix broken links.
6. Improve the main app or main user action.
7. Add support pages.
8. Add topic pages.
9. Add business planning later.
10. Commit each useful safe change.
11. Continue through the safe queue until blocked.
12. Report only after several useful commits or a real blocker.

## Stop points

Stop only for blocked writes, major code deletion, framework changes, private keys, payment setup, live tracking, live ads, unclear repository state, or a change that could break production in a way that cannot be safely repaired in the same work session.

## Tech rule

Keep simple website repos on HTML, CSS, and JavaScript unless the owner clearly requests a rebuild.

Do not switch to React, Vite, Next.js, TypeScript, or build tools unless that repo already uses them and its locked checkpoint allows it.
